# Holo Image 新实例收敛与 Admin WebUI 设计

日期：2026-03-29  
仓库：`C:\Users\HeLuo\.config\superpowers\worktrees\source\feat-text-chat-compat`

## 1. 背景

当前项目已经存在两个线上实例：

- 旧实例：`https://holo-image-api.zeabur.app`
- 新实例：`https://holo-image-api-eners.zeabur.app`

两者都具备自动注册、自动补号、自动清理的基础能力，但只有新实例已经验证了真正的持久化与重启恢复能力：

- 新实例 `restore_loaded > 0`
- 旧实例 `restore_loaded = 0`

这意味着旧实例依然是“可写但不持久”的半成品，而新实例才是能长期承载服务的目标实例。

用户已经明确要求：

1. 所有能力全部迁移到新实例
2. 最终对外域名仍使用 `https://holo-image-api.zeabur.app`
3. 老实例在完成迁移后退役
4. 提供一个可直接管理的新实例 Admin WebUI
5. WebUI 复用现有 admin key，不另造账号密码体系

## 2. 目标

本次工作的目标是把系统收敛成“一个真正长期可用的线上实例”：

1. 新实例成为唯一真实服务实例
2. 旧实例当前池中账号尽可能迁移到新实例
3. 旧域名切换到新实例
4. Admin WebUI 直接挂在新实例下提供管理能力
5. 迁移完成后可以下线老实例，不再依赖本地导入或双实例共存

## 3. 非目标

以下内容不在本次范围内：

1. 不新增完整的账号系统、用户名密码登录、RBAC
2. 不做复杂的多租户后台
3. 不做双实例长期并行的统一运营面板
4. 不在浏览器侧暴露 JWT 明文列表
5. 不做与当前迁移目标无关的 UI 花活

## 4. 当前状态判断

### 4.1 旧实例

旧实例目前的优点是池子更大、跑得更久，但核心缺陷是：

- `restore_loaded = 0`
- 无法作为最终承载实例

因此旧实例只能作为过渡来源，不应该继续承担最终域名职责。

### 4.2 新实例

新实例已经证明具备：

- 自动注册
- 自动补号
- 自动 prune
- 持久化账号仓库
- 重启恢复

因此新实例是本次收敛的目标实例。

## 5. 选型结论

采用“**单服务内置 Admin WebUI + 服务端迁移接口 + 最终域名切换**”方案。

原因：

1. 部署面最小，避免再起一个前端服务
2. 所有真相源都收敛到新实例
3. 管理页、迁移逻辑、池状态、持久化状态全部在同一服务里
4. 最符合“最终只保留一个实例”的目标

## 6. 设计概览

### 6.1 最终形态

最终服务形态如下：

- 唯一线上实例：新实例
- 唯一对外域名：`https://holo-image-api.zeabur.app`
- 唯一后台入口：`https://holo-image-api.zeabur.app/admin`
- 唯一管理鉴权：现有 `Authorization: Bearer <admin key>`

### 6.2 迁移阶段形态

迁移期间保留：

- 旧实例继续对外运行，避免突然掉线
- 新实例持续补号与持久化
- 通过服务端迁移把旧实例池子搬到新实例

当新实例完成迁移且健康后，再切旧域名。

## 7. 组件设计

### 7.1 Admin WebUI

WebUI 作为内置静态页面，由新实例直接提供。

新增路由：

- `GET /admin`
- `GET /admin/assets/*`

页面风格采用“产品后台外观 + 运维控制能力”的混合方案：

- 深色主题
- 卡片化状态区
- 操作按钮集中
- 关键危险操作独立分区

### 7.2 Admin API

保留并复用现有接口：

- `GET /v1/admin/pool`
- `POST /v1/admin/pool/fill`
- `POST /v1/admin/pool/prune`
- `POST /v1/admin/pool/import`

新增接口：

- `GET /v1/admin/meta`
  - 返回当前实例名称、域名、是否主实例、当前版本/提交信息、迁移状态

- `POST /v1/admin/migrate-from-old`
  - 服务端从旧实例拉取账号并导入新实例
  - 导入后自动落盘
  - 返回迁移统计信息

- `GET /v1/admin/migration/status`
  - 返回最近一次迁移结果与当前收敛状态

- `POST /v1/admin/retire-old`
  - 标记迁移完成后的老实例退役动作
  - 第一版只负责调用退役流程或输出明确失败，不做静默成功

### 7.3 旧实例导出接口

为了让迁移不在浏览器里暴露 JWT，旧实例增加一个仅 admin key 可访问的导出接口：

- `GET /v1/admin/pool/export`

返回内容：

- 当前内存池中的 JWT 列表
- 可选的 quota 信息
- 不返回无关敏感信息

该接口只作为迁移期使用。迁移完成后可以保留但默认不在 UI 中显式展示。

### 7.4 持久化仓库

继续使用已经实现的文件型账号仓库：

- `POOL_STORE_PATH=/app/data/holo-image-api/pool-accounts.json`

规则：

1. 自动注册成功后立即落盘
2. import 后立即落盘
3. prune 删除失效账号后同步落盘
4. 启动时自动恢复

## 8. WebUI 页面结构

第一版页面包含 6 个模块：

### 8.1 总览

- 当前池子数量
- 低水位
- 自动补号开关状态
- 当前注册并发
- 注册成功/失败
- prune 次数

### 8.2 持久化状态

- `persistence_enabled`
- `persisted_count`
- `restore_loaded`
- `restore_rejected`
- `last_persist_at`
- `last_restore_at`

### 8.3 池管理

- 手动 fill
- 手动 prune
- 手动刷新状态

### 8.4 迁移中心

- 显示旧实例当前状态摘要
- “迁移旧实例池到新实例”按钮
- 迁移结果卡片：
  - requested
  - imported
  - duplicates
  - rejected
  - total_count

### 8.5 域名切换状态

- 当前主域名目标
- 是否已切到新实例
- 是否满足切换条件

### 8.6 危险操作

- 退役老实例
- 需要二次确认
- 明确显示该动作不可逆或高影响

## 9. 迁移流程

### 阶段 1：准备

1. 新实例继续自动补号
2. 新实例继续持久化
3. 新实例 WebUI 与迁移接口上线

### 阶段 2：迁移旧池

1. 新实例调用旧实例 `/v1/admin/pool/export`
2. 服务端过滤重复 JWT
3. 调用新实例内部导入逻辑
4. 导入后立即落盘
5. 记录最近一次迁移结果

### 阶段 3：验证

满足以下条件后，允许切旧域名：

1. 新实例迁移动作成功执行
2. 新实例 `restore_loaded > 0`
3. 新实例池子达到安全阈值
4. 新实例健康检查正常

### 阶段 4：切流量

1. 将 `https://holo-image-api.zeabur.app` 切到新实例
2. 再次验证：
   - `/healthz`
   - `/v1/models`
   - `/v1/admin/pool`
3. 确认成功后，准备退役老实例

### 阶段 5：退役

1. 老实例停止承担对外请求
2. 仅保留短暂观察窗口
3. 观察通过后彻底退役

## 10. 鉴权与安全

### 10.1 鉴权

WebUI 第一版复用现有 admin key：

- 前端通过输入或保存 admin key 调用 admin API
- 服务端仍用 Bearer 鉴权中间件

### 10.2 JWT 暴露控制

默认策略：

1. WebUI 不显示 JWT 明文
2. 浏览器不直接拉旧实例导出接口
3. 迁移动作由新实例服务端执行

### 10.3 错误暴露

坚持当前工程原则：

- 不静默 fallback
- 不伪装成功
- 所有失败直接显示到 WebUI

## 11. 错误处理

### 11.1 迁移失败

显示明确错误：

- 旧实例不可达
- admin key 校验失败
- 导出为空
- 导入全部重复
- 导入后落盘失败

### 11.2 域名切换失败

显示：

- 当前域名绑定状态
- 目标 service
- 切换失败原因

### 11.3 持久化异常

通过现有状态字段直接展示：

- `last_persist_error`
- `persistence_enabled`

## 12. 测试与验证

### 12.1 单元测试

新增/扩展测试覆盖：

1. 旧实例导出接口鉴权与返回格式
2. 新实例迁移接口
3. WebUI 静态路由
4. 域名切换状态判定逻辑
5. 危险操作确认逻辑

### 12.2 集成验证

至少验证：

1. 新实例自动注册仍然工作
2. 新实例 prune 仍然工作
3. 新实例持久化仍然工作
4. 新实例重启后恢复仍然工作
5. 迁移后新实例池数量上升
6. 域名切换后旧域名正常命中新实例

## 13. 回滚策略

若切旧域名后出现问题：

1. 旧域名重新指回老实例
2. 新实例保留数据，不清空
3. 重新分析失败点后再重试切换

## 14. 成功标准

本次设计只有在以下条件全部满足时才算完成：

1. 新实例能独立自动补号
2. 新实例能独立自动 prune
3. 新实例能独立持久化
4. 新实例重启后 `restore_loaded > 0`
5. 老实例池能迁到新实例
6. `https://holo-image-api.zeabur.app` 最终指向新实例
7. WebUI 可以完成日常管理
8. 老实例可以安全退役
